   The error `ERROR: unrecognized option 'no-available'` was caused by an invalid flag `--no-available` in the `apk add` command. The correct flag for disabling the index cache in Alpine is `--no-cache`.

   ```dockerfile

   RUN apk add --no-cache openssl libc6-compat
   ```

   - Use `libc6-compat`: Alpine uses `musl` instead of `glibc`. Prisma's query engine often requires `libc6-compat` to bridge this gap on Alpine.
   - Use `openssl`: Prisma requires OpenSSL to establish secure database connections.
   - Multi-stage builds: Always use multi-stage builds to keep the final production image lean.
   - Node Version: Ensure Node.js 20+ is used for Prisma v7 compatibility.

4. Verified Fix:
   - Removed `--no-available` flag.
   - Added `libc6-compat` for enhanced binary compatibility.
   - Optimized `apk` command to use `--no-cache` for smaller image layers.

💾 PRISMA DOCKER FIX: MISSING DATABASE_URL IN SCHEMA
================================================================================
1. Why this error happens:
   Prisma requires a `DATABASE_URL` to connect to your database. When running `prisma migrate deploy` (especially inside a Docker container), Prisma looks for the `url` property in your `schema.prisma` file. If that property is missing or doesn't correctly point to an environment variable using `env("DATABASE_URL")`, the command fails because it has no destination to apply migrations to.

2. Correct DATABASE_URL Format for Docker Compose:
   Inside a Docker network, services communicate using their **service names** as hostnames. 
   Format: `postgresql://USER:PASSWORD@SERVICE_NAME:PORT/DB_NAME`
   Example: `postgresql://user:password@db:5432/saas_db`

3. Correct prisma/schema.prisma configuration:
   ```prisma
   datasource db {
     provider = "postgresql"
     url      = env("DATABASE_URL")
   }
   ```

4. Working Backend Service Example (docker-compose.yml):
   ```yaml
   backend:
     build: ./server
     environment:
       - DATABASE_URL=postgresql://user:password@db:5432/saas_db
     depends_on:
       db:
         condition: service_healthy
   ```

5. Best Practices for Environment Variables:
   - Use a `.env` file for local development (ensure it's in `.gitignore`).
   - In `docker-compose.yml`, you can use `env_file: .env` to load all variables at once.
   - For production, pass sensitive secrets (like DB passwords) via your CI/CD provider's secret manager or Docker Secrets.
    - Always wait for the database to be "healthy" before starting the backend to avoid connection-refused errors.

================================================================================
1. Why this error happens:
   In Prisma v7, even if you have a `prisma.config.ts` for CLI tools (like `migrate`), the `PrismaClient` instantiated in your application code sometimes fails to automatically pick up the connection URL if it's missing from `schema.prisma`. This results in the `PrismaClientInitializationError: PrismaClient needs to be constructed with a non-empty, valid PrismaClientOptions`.

2. Corrected PrismaClient Initialization:
   The client must be explicitly initialized with the `DATABASE_URL` at runtime to ensure it correctly connects to the PostgreSQL container in the Docker network.

   ```typescript

     datasources: {
       db: { url: process.env.DATABASE_URL },
     },
   });
   ```

3. Compatibility Note:
   While `prisma.config.ts` handles the CLI (migrations, generation), the runtime application code still requires the environment variable to be passed into the `PrismaClient` constructor for maximum reliability in Docker environments.
