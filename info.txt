💾 PRISMA DOCKER FIX: MISSING DATABASE_URL IN SCHEMA
================================================================================
1. Why this error happens:
   Prisma requires a `DATABASE_URL` to connect to your database. When running `prisma migrate deploy` (especially inside a Docker container), Prisma looks for the `url` property in your `schema.prisma` file. If that property is missing or doesn't correctly point to an environment variable using `env("DATABASE_URL")`, the command fails because it has no destination to apply migrations to.

2. Correct DATABASE_URL Format for Docker Compose:
   Inside a Docker network, services communicate using their **service names** as hostnames. 
   Format: `postgresql://USER:PASSWORD@SERVICE_NAME:PORT/DB_NAME`
   Example: `postgresql://user:password@db:5432/saas_db`

3. Correct prisma/schema.prisma configuration:
   ```prisma
   datasource db {
     provider = "postgresql"
     url      = env("DATABASE_URL")
   }
   ```

4. Working Backend Service Example (docker-compose.yml):
   ```yaml
   backend:
     build: ./server
     environment:
       - DATABASE_URL=postgresql://user:password@db:5432/saas_db
     depends_on:
       db:
         condition: service_healthy
   ```

5. Best Practices for Environment Variables:
   - Use a `.env` file for local development (ensure it's in `.gitignore`).
   - In `docker-compose.yml`, you can use `env_file: .env` to load all variables at once.
   - For production, pass sensitive secrets (like DB passwords) via your CI/CD provider's secret manager or Docker Secrets.
    - Always wait for the database to be "healthy" before starting the backend to avoid connection-refused errors.

🎓 INTERVIEW PREPARATION: TECHNICAL DEEP DIVE
================================================================================
Q1: How do you handle database consistency for financial operations?
A: I use Prisma's `$transaction` API. For example, when creating a transaction, we simultaneously check budgets and create audit logs. If any step fails, the entire operation rolls back. This prevents partial writes and ensures financial data integrity.

Q2: Explain your JWT Refresh Token Rotation strategy.
A: To prevent token theft, I implement rotation. When a user refreshes their access token, I issue a NEW refresh token and revoke the old one. If the system detects a used/revoked token being presented, it assumes a breach and revokes ALL active sessions for that user. This provides a high level of security against session hijacking.

Q3: How does your system scale to handle 10,000 concurrent users?
A: 
1. Database Indexing: I've indexed `organizationId` and `createdAt` to ensure O(log N) query performance.
2. Caching: Redis stores dashboard summaries, reducing DB load for frequently accessed data.
3. Connection Pooling: Using Prisma's built-in pooling to manage database connections efficiently.
4. Stateless Backend: The backend is stateless (JWT-based), allowing it to be horizontally scaled across multiple containers behind a load balancer.

Q4: How do you prevent N+1 query problems in your API?
A: I use Prisma's `include` and `select` features which perform efficient SQL JOINs or batched queries. For example, when fetching transactions, I include the `tags` relation in a single database round-trip.

Q5: What is your strategy for handling large data exports?
A: I use Cursor-Based Streaming. Instead of loading 100k rows into memory (which would crash the server), I fetch data in batches of 1000 and stream them directly to the HTTP response using `fast-csv`. This maintains a low and constant memory footprint.
